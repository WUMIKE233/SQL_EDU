# 数据定义与完整性约束

## 2.1 建表之前先建模

在写 `CREATE TABLE` 之前，应先回答三个问题：

1. 业务中有哪些对象需要保存？
2. 每个对象有哪些属性？
3. 对象之间是什么关系？

以教学管理系统为例：

- 一个院系有多名教师和学生
- 一位教师可以讲授多门课程
- 一位学生可以选多门课程
- 一门课程也可以被多位学生选择

## 2.2 常见数据类型

| 类型 | 用途示例 |
| --- | --- |
| `INTEGER` | 编号、学年 |
| `VARCHAR(n)` | 姓名、专业、课程名 |
| `DATE` | 入学日期、开课日期 |
| `NUMERIC(p, s)` | 学分、成绩、金额 |
| `BOOLEAN` | 是否必修 |

选择数据类型时，应遵循“足够表达，但不过度浪费”的原则。

## 2.3 创建表

```sql
CREATE TABLE departments (
    department_id INTEGER PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL UNIQUE,
    office_phone VARCHAR(30)
);
```

这张表表达了三个信息：

- `department_id` 用于唯一标识院系
- `department_name` 不能为空，且不允许重名
- `office_phone` 可以为空

## 2.4 约束的作用

约束用于保证数据质量。与“靠程序员自觉”相比，数据库层面的约束更可靠。

常见约束包括：

- `PRIMARY KEY`：主键约束
- `NOT NULL`：非空约束
- `UNIQUE`：唯一约束
- `CHECK`：检查约束
- `FOREIGN KEY`：外键约束
- `DEFAULT`：默认值

## 2.5 多表与外键

```sql
CREATE TABLE students (
    student_id INTEGER PRIMARY KEY,
    student_name VARCHAR(100) NOT NULL,
    gender VARCHAR(10) CHECK (gender IN ('男', '女')),
    enroll_year INTEGER NOT NULL CHECK (enroll_year >= 2020),
    department_id INTEGER NOT NULL,
    major VARCHAR(100) NOT NULL,
    email VARCHAR(120) UNIQUE,
    CONSTRAINT fk_students_department
        FOREIGN KEY (department_id)
        REFERENCES departments(department_id)
);
```

外键的意义在于：

- 保证学生所属院系必须存在
- 防止“孤立数据”
- 明确业务关系

## 2.6 修改表结构

```sql
ALTER TABLE students
ADD COLUMN birth_date DATE;
```

```sql
ALTER TABLE students
ALTER COLUMN email SET NOT NULL;
```

修改表结构时要特别注意已有数据是否满足新规则。比如，如果表中已经存在空邮箱记录，那么直接加 `NOT NULL` 会失败。

## 2.7 删除对象

```sql
DROP TABLE enrollments;
```

删除对象是高风险操作。教学和生产中都应先确认：

- 是否存在依赖对象
- 是否已经备份
- 是否会影响他人使用

## 2.8 本章小结

- 建表前先做概念建模
- 约束不是“可选项”，而是保证数据正确性的关键机制
- 外键帮助我们在数据库层面维护表与表之间的关系
- 修改与删除结构时，应先评估影响范围
